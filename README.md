# DirectTransfer

Windows 点对点文件传输工具。使用 C++17、Win32 GUI 和自定义 Winsock TCP 协议，不依赖 SMB、Robocopy、网络共享或 Windows 凭据。

## 使用流程

1. 两台计算机使用直连网线连接，或接入同一局域网。
2. 接收端选择目标目录并开始监听。
3. 发送端选择源目录，通过自动发现或手动 IP 连接接收端。
4. 确认 6 位配对码后开始传输。
5. UI 显示精确文件数、总大小、已提交进度、速度、已用时间和 ETA。

手动 IP 和 APIPA 地址可以直接使用，不要求管理员权限。管理员静态 IP 配置仅是可选辅助功能。

## 传输流程

```text
快速盘点源目录
→ 固定总文件数和总字节数
→ 分块发送二进制文件目录
→ 小文件窗口批流 / 大文件分块流
→ 接收端写入 .dtpart 并原子提交
→ 批次或文件提交确认
→ 会话最终提交确认
```

- 小于等于 64 KiB 的文件最多 512 个组成一个批次。
- 最多保持 4 个小文件批次在途，源端读取、网络发送和目标端写盘可以重叠。
- 小文件写入线程数根据目标介质的寻道特性选择：旋转磁盘 1 个，SSD 4 个，未知或网络路径 2 个。
- 文件内容在读取和写入时使用 CRC32 校验，不进行传输前的第二次完整文件读取。
- 进度按接收端已提交的文件和字节统计；重传流量不会重复增加主进度。

## 中断恢复边界

- TCP 连接中断后，发送端自动指数退避重连，无需手动点击。
- 接收端进程仍在运行时，已提交文件不会重传；未提交的小文件批次或大文件从头重传。
- 当前版本不实现大文件内部字节偏移续传。
- 用户取消、接收端进程退出、机器关机、磁盘永久错误不属于自动会话恢复范围。

协议通过 `TransferId`、目录 CRC、`FileId`、`BatchId` 和能力列表保留后续扩展持久化检查点、文件内断点、增量、镜像策略及更强校验算法的空间。

## 网络协议

- UDP 端口 `49321`：局域网自动发现。
- TCP 端口 `49321`：握手、目录、文件和提交确认。
- 包格式：`1 字节类型 + 4 字节小端载荷长度 + 载荷`。
- 当前协议主版本：`5`。
- 配对阶段使用有限超时；传输阶段按 5 秒 I/O 等待周期检测进展，连续 90 秒无进展后进入自动重连。

## 构建

```powershell
cmake -G Ninja -DCMAKE_CXX_COMPILER=g++ -B build
cmake --build build
.\build\DirectTransfer.exe
```

依赖：MinGW-W64 g++ 16.1、CMake 4.3.2、Ninja 1.13.2、nlohmann/json v3.11.3。

## 系统行为

- 不创建网络共享，不调用 SMB，不修改 Windows 凭据或 LSA 策略。
- 不自动添加防火墙规则。
- 接收文件先写入 `.dtpart`，校验成功后原子替换最终文件。
- 日志和报告使用 UTF-8。
