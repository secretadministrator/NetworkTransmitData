# DirectTransfer 变更记录

> 说明：本文档基于当前仓库源码、`AGENTS.md` 约束与现有构建配置整理，用于沉淀项目的代码迭代、问题分析与解决方案。后续新增改动请继续按“背景 -> 问题 -> 方案 -> 验证”补充，保持可追溯性。

## 当前版本基线

- 项目定位：Windows 直连点对点文件传输工具，使用 Win32 GUI + Winsock TCP，自定义协议，不依赖 SMB、Robocopy、网络共享或 Windows 凭据。
- 技术栈：C++17、Win32 API、CMake + Ninja、MinGW-W64 g++、nlohmann/json。
- 目录结构：`src/` 下按“配置、界面、网络、协议、计划、续传、进度、报告、网络配置”拆分为独立模块。

## 主要迭代改动

### 1. 从批处理 Robocopy 流程迁移到原生程序

- 变更内容：
  - 旧的 `.bat` 直连脚本仍保留在仓库中作为历史参照。
  - 新主线改为 `DirectTransfer.exe`，由 C++17 Win32 GUI 统一承载发送端/接收端流程。
- 问题分析：
  - 批处理方案对系统环境、共享、权限和命令参数依赖重，交互不统一，错误处理也不易维护。
  - 复杂传输场景下，单脚本难以稳定支持清单、续传、完整性校验和状态回显。
- 解决方案：
  - 使用 `MainWindow`、`RoleSelectPage`、`SenderPage`、`ReceiverPage` 构建分步式 GUI。
  - 把网络、协议、计划、续传、日志拆成独立模块，降低耦合度。

### 2. 自定义 TCP 协议替代 SMB / 共享访问

- 变更内容：
  - `TransferProtocol` 定义 `1 字节类型 + 4 字节小端长度 + payload` 的二进制包格式。
  - 传输端口统一为 `49321`，通过 UDP 发现和 TCP 会话完成握手、清单、计划和文件传输。
- 问题分析：
  - SMB / 共享路径方案对凭据、组策略和系统共享配置依赖高，不适合轻量直连工具。
  - 通用文本协议在大文件和二进制内容下容易出现解析歧义。
- 解决方案：
  - 采用固定头部长度的二进制协议，明确区分 `MANIFEST`、`TRANSFER_PLAN`、`FILE_HEADER`、`FILE_CHUNK`、`FILE_DONE`、`FILE_DONE_ACK`、`ERROR_MSG` 等消息。
  - 用专用包解析器 `Packet::Serialize / Deserialize` 保证收发一致。

### 3. 配对认证与发现流程

- 变更内容：
  - `PairingHandler` 负责 6 位配对码生成与验证。
  - `NetworkDiscovery` 负责局域网发现，发送端可自动发现接收端。
- 问题分析：
  - 仅依赖 IP 直连会增加手工输入成本，也容易连错目标。
  - 若没有轻量认证，局域网内任意主机都可能接入。
- 解决方案：
  - 用短期配对码完成会话认证，避免引入 Windows 凭据体系。
  - 在界面上保留可见配对码和手动 IP 两种路径，兼顾易用性与可控性。

### 4. 清单、计划与传输模式分离

- 变更内容：
  - `Manifest` 只负责文件清单 JSON：路径、大小、SHA-256。
  - `TransferPlanner` 根据目标目录和模式生成 `TRANSFER / SKIP / OVERWRITE` 计划。
  - `TransferMode` 支持 `SAFE_COPY`、`OVERWRITE`、`MIRROR`。
- 问题分析：
  - 直接发送文件会让接收端无法提前决定跳过、覆盖或清理。
  - 如果把“计划”和“传输”混在一起，镜像、增量、覆盖三种语义会互相干扰。
- 解决方案：
  - 先发清单，再由接收端构建计划，明确每个文件的动作和剩余字节数。
  - 镜像模式下先删除目标端多余文件，再进入正式接收，减少残留脏数据。

### 5. 续传与断点完整性增强

- 变更内容：
  - `ResumeManager` 以 `.dtpart` 作为临时文件后缀，检测可续传偏移。
  - 计划中加入 `offset` 和 `resumeHash`，用于验证断点前缀是否与源文件一致。
  - 发送端对残余字节进行精确发送，不越界、不补发多余数据。
- 问题分析：
  - 仅凭文件大小续传会在中途修改、文件截断或旧临时文件残留时产生错误恢复。
  - 如果断点前缀和源文件不一致，继续续传会导致最终文件损坏。
- 解决方案：
  - 对 `.dtpart` 前缀计算 SHA-256，前缀不匹配时强制回退到 `offset = 0`。
  - 接收端使用 `.dtpart` 先落盘，完成后再重命名，避免半成品污染正式文件。

### 6. 完整性校验与错误处理

- 变更内容：
  - `Manifest` 和 `TransferPlanner` 都携带 `sha256` 信息，`SKIP` 判断必须同时满足大小和哈希一致。
  - 发送端与接收端对 `OPEN_FAILED`、`READ_FAILED`、`HASH_FAILED`、`TRANSFER_ABORTED` 等错误进行显式传递。
  - `FILE_DONE_ACK` 采用路径绑定格式，防止跨文件串包。
- 问题分析：
  - 仅比较文件大小无法确认内容一致。
  - 传输中任意一侧出错后，如果协议未对齐，后续文件会整体失步。
- 解决方案：
  - 通过文件级 SHA-256 保证跳过决策和接收结果可验证。
  - 接收端遇到写入失败时仍然继续 drain 当前文件流，再回 `FAIL`，维持协议边界对齐。
  - 发送端对缺失、错配或失败的 ACK 视为会话级错误，不继续后续文件。

### 7. 传输稳定性与超时策略

- 变更内容：
  - 握手阶段保留有限超时。
  - 进入实际文件传输和校验后，不再沿用固定 30 秒 socket 超时。
- 问题分析：
  - 大文件、慢速磁盘或局域网抖动下，固定短超时会误判为失败。
  - 传输阶段和配对阶段的时延特征不同，应该采用不同策略。
- 解决方案：
  - 把超时约束限定在配对和握手前置阶段。
  - 进入正式传输后清除旧超时，让数据流按协议节奏完成。

### 8. 进度、日志与报告体系

- 变更内容：
  - `ProgressTracker` 基于最近 10 个采样计算速度，提供剩余时间估算。
  - `ProgressControl` 负责进度条与文本状态同步显示。
  - `AuditLogger` 与 `ReportGenerator` 统一使用 UTF-8 窄流写日志和报告。
- 问题分析：
  - 如果进度仅显示已传字节，没有速度和 ETA，用户很难判断传输是否正常。
  - 日志和报告若使用宽流或混合编码，跨工具查看时容易乱码。
- 解决方案：
  - 统一进度采样和展示逻辑，补齐速度、文件数和当前文件信息。
  - 记录文件和报告采用 UTF-8，避免编码回退导致的不可读问题。

### 9. 可选 IP 配置与 DHCP 恢复

- 变更内容：
  - `IpConfigurator` 提供网卡检测、静态 IP 配置和 DHCP 恢复。
  - `main.cpp` 在程序退出时按配置状态尝试恢复 DHCP。
- 问题分析：
  - 直连场景常常需要临时静态 IP，但不应把这一步变成强制前置条件。
  - 配置一旦改动，退出后需要可逆，否则会干扰用户现有网络环境。
- 解决方案：
  - 保留手动 IP 和直连接收主流程，不把管理员 IP 配置做成硬性步骤。
  - 通过退出恢复逻辑把网络状态尽量还原。

### 10. 构建、资源与兼容性

- 变更内容：
  - CMake 自动下载 `nlohmann/json` 单头文件。
  - 使用资源文件保留 `VERSIONINFO`。
  - 链接 `ws2_32`、`iphlpapi`、`advapi32`、`ole32` 等必要系统库，并保持静态链接。
- 问题分析：
  - 依赖外部手工放置头文件会降低可复现性。
  - 缺少版本信息和资源文件，会影响程序属性展示与兼容性。
- 解决方案：
  - 在 `CMakeLists.txt` 中自动补齐依赖下载逻辑。
  - 保留资源与版本信息，减少发行和兼容问题。

## 当前实现约束

- 不使用 SMB、Robocopy、共享凭据或注册表改写作为主流程。
- 不自动创建防火墙规则，不把管理员权限作为所有流程的前置条件。
- 传输协议必须保证完整收发，`SendPacket` 需要循环发送直到全部写完。
- 传输过程中不得重新引入“先收完整文件再二次全量校验”的低效路径。
- UI 字符串和日志输出保持中文可读、UTF-8 友好。

## 验证状态

- 构建方式：`cmake -G Ninja -DCMAKE_CXX_COMPILER=g++ -B build`，随后 `cmake --build build`。
- 当前仓库未见自动化测试与 CI 脚本，仍以手工构建和运行验证为主。
